## 2026-05-05 · 项目启动

### 需求分析

用户需要一个桌面 GUI 工具，自动抓取中国移动资费公示页面的套餐数据。核心要求：
- 浏览器可视化，用户手动选省份
- 自动遍历所有资费类型
- 18 列完整字段，含 4 个新增字段
- 支持搜索筛选和 Excel 导出

### 技术选型

- **GUI**：PySide6（Qt for Python，跨平台，界面美观）
- **浏览器控制**：Selenium（成熟稳定，社区活跃）
- **Excel 导出**：openpyxl（纯 Python，不依赖 COM）

## 踩坑记录 1：ChromeDriver 下载失败

### 问题

```
Message: Unable to obtain driver for chrome
```

Selenium 4.6+ 内置的 ChromeDriver 自动下载需要访问 `googlechromelabs.github.io`，国内网络无法访问。

### 尝试的方案

| 方案 | 结果 |
|------|------|
| Selenium 内置管理 | 失败，需要翻墙 |
| webdriver-manager 库 | 失败，VM 环境网络受限 |
| 子进程 + 调试端口 | 失败，Selenium 连接仍需 ChromeDriver |
| 手动指定 Chrome 路径 | 失败，ChromeDriver 版本不匹配 |

### 最终方案

实现多级回退策略：
1. 检查程序同目录是否有 `chromedriver.exe`
2. 检查本地缓存
3. 通过 Google API 查询匹配版本号 → 从 npmmirror 国内镜像下载
4. 回退到 Selenium 内置管理

关键代码：通过 `googlechromelabs.github.io/chrome-for-testing/latest-patch-versions-per-build-with-downloads.json` API 查询 Chrome 版本对应的精确 ChromeDriver 版本号，再用国内镜像下载。

## 踩坑记录 2：资费类型下拉框操作

### 问题

页面没有原生 `<select>` 元素，是 Vue 自定义下拉组件，无法用标准 Selenium select 操作。

### 分析过程

通过 `browser_evaluate` 执行 JS 分析 DOM，发现：
- 下拉框容器：`div.select-container.the-select`
- 展开选项：`div.select-list-box > div.select-item`
- 选项值：`data-value` 属性
- 选项文本：`span.item-label`

### 解决方案

用 Selenium 点击 `.select-box` 展开下拉框，然后遍历 `.select-item` 读取选项，点击目标选项切换。

## 踩坑记录 3：滚动策略

### 问题

最初采用逐步滚动（每次滚 50%~80% 视口），但页面是**滚到底部才触发懒加载**，逐步滚动永远无法触发。

### 尝试的方案

| 方案 | 结果 |
|------|------|
| 每次滚 80% 视口 | 无法触发加载 |
| 每次滚 50% 视口 | 无法触发加载 |
| **每次直接滚到底部** | **成功触发加载** |

### 最终方案

每轮直接 `scrollTo(0, document.body.scrollHeight)`，等待 1 秒后检查是否有新数据。

## 踩坑记录 4：UI 卡顿

### 问题

拖动窗口时界面无响应，抓取越久越卡。

### 原因分析

DOM 解析在主线程执行。Selenium 的 `find_elements()` 和 `element.text` 调用会阻塞 UI 线程。数据量越大，每次解析遍历的元素越多，UI 越卡。

### 解决方案

**将所有 DOM 操作移到爬虫线程**，主线程只通过信号接收解析后的纯 Python 列表数据。

```
优化前：主线程 → find_elements() → card.text → parse → 更新表格
优化后：子线程 → JS提取 → parse → 信号发送 → 主线程只做表格更新
```

## 踩坑记录 5：抓取越来越慢

### 问题

前几轮每轮 2 秒，到 100 条时每轮 8 秒，到 200 条时每轮 15 秒。

### 原因分析

CSS 选择器 `[class*='tariff'] [class*='item']` 匹配了 1117 个嵌套子元素（每个卡片内部也有带这些 class 的子元素）。每次解析都要遍历 1117 个元素调用 `.text`。

### 解决方案

1. **精确选择器**：通过分析 DOM 找到 `.tariff-item-container`，每个套餐精确一个 div
2. **JS 批量提取**：用 `execute_script` 一次性提取所有卡片的 `innerText`，替代逐个 `element.text`
3. **纯 Python 解析**：拿到文本后用正则解析，不再调用 Selenium

```
优化前：1117个Selenium元素 → 逐个.text → 逐个parse → O(n²)
优化后：1次JS调用 → N个字符串 → 正则parse → O(n)
```

## 踩坑记录 6：套餐名称提取错误

### 问题

部分卡片的套餐名称显示为 "0元/月"，和资费标准一样。

### 原因分析

原来的逻辑是取文本中第一个不包含排除关键字的行作为套餐名称。"0元/月" 不包含任何排除关键字（资费标准、适用地区等），被误判为套餐名称。

### 解决方案

用 JS 直接提取 `.item-name` 元素的文本作为套餐名称，不再靠文本猜测。

```javascript
var nameEl = cards[i].querySelector('.item-name');
name: nameEl ? nameEl.textContent.trim() : ''
```

## 性能优化总结

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 每轮解析时间 | 随数据量线性增长 (10秒+) | 恒定 (~0.3秒) |
| UI 响应 | 卡顿/无响应 | 流畅 |
| CSS 选择器匹配 | 1117 个元素 | 精确匹配实际卡片数 |
| 套餐名称准确率 | 有误判 | 100% 精确 |
| 判定到底 | 页面高度（不可靠） | 卡片数量（精确） |
| ChromeDriver | 需翻墙下载 | 国内镜像自动下载 |

---

## 关键技术决策

1. **可视化浏览器**：不使用无头模式，用户可以观察抓取过程、手动操作页面
2. **JS 批量提取**：1 次 `execute_script` 替代 N 次 `element.text`，性能提升数十倍
3. **线程分离**：GUI 和爬虫完全分离，通过信号通信，UI 永不卡顿
4. **精确选择器**：`.tariff-item-container` 直接定位套餐卡片
5. **卡片数量判断**：用 JS 快速获取卡片数（毫秒级），只在数量变化时才做完整解析
6. **多级回退**：ChromeDriver 获取有多种方式，确保在各种环境下都能运行
